# rdit

`rdit` installs coding-agent audit hooks and provides a TUI for consuming the generated reasoning audits.

## Runtime layout

- `.rdit/`
  Local runtime directory managed by `rdit`.
- `.rdit/audits_reports.db`
  SQLite database used for source tracking and processed findings.
- `.rdit/judge_output_schema.json`
  Shared schema materialized for structured judge output.
- `reasoning_audits/`
  Audit markdown files generated by the installed hooks and skills.

`rdit init` updates `.gitignore` so `.rdit/` and `reasoning_audits/` stay local.

## Storage model

SQLite uses two main tables:

- `audit_sources`
  One row per immutable file in `reasoning_audits/`.
  Tracks file path, hash, processing state, and selected judge provenance.
- `audit_findings`
  Zero or more finding rows per source file.
  Stores the board fields: title, issue, why, how, and score.

The judge contract is zero-to-many findings per source file.
If a source file has no meaningful issues, the source row is still marked processed and no finding row is created.

Raw provider output is not persisted.
`rdit` parses the structured response and stores only the normalized fields needed for the board and detail view.

## Judge providers

`rdit` currently supports two headless judge runners:

- Codex
  Uses `codex exec --ephemeral --model ... --output-schema ... --output-last-message ...`
- Claude Code
  Uses `claude --print --model ... --json-schema ... --no-session-persistence --output-format json`

The TUI lets the user choose the judge provider and model independently from the provider that originally produced the audit log.

## Concurrency

Audit judging uses bounded goroutine concurrency for external provider calls.
SQLite writes remain serialized through the storage layer.
The database is opened with WAL mode, a busy timeout, and a single-connection pool to reduce lock contention.
